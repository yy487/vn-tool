# ONI Engine 资源 Dump 工具 - 使用指南

## 文件清单

```
oni_winmm.c      ← DLL源码（winmm.dll劫持 + 图像dump）
build_msvc.bat   ← MSVC 一键编译脚本
build_mingw.bat  ← MinGW 一键编译脚本
gr2_tool.py      ← GR2 图像格式转换工具（独立使用，不依赖DLL）
```

---

## 方案选择

| 需求 | 推荐方案 |
|------|----------|
| 只需要解码文件系统上的 .gr2 文件 | 用 `gr2_tool.py`，不需要DLL |
| 需要dump游戏运行时加载的所有图像（含oni.dat内嵌） | 用 `oni_winmm.c` 编译DLL |
| 汉化替换图片 | `gr2_tool.py encode` 或直接放 .bmp（引擎自动回退） |

---

## 编译 DLL

### 方法 A：MSVC（推荐）

**前提**：安装了 Visual Studio（任何版本，社区版免费）

1. 打开 **开始菜单** → **Visual Studio** → **Developer Command Prompt**
   （或 "x86 Native Tools Command Prompt"——必须是 **x86**，游戏是32位的）

2. cd 到源码目录，运行：
```cmd
build_msvc.bat
```

或手动：
```cmd
cl /LD /O2 oni_winmm.c /Fe:winmm.dll /link user32.lib kernel32.lib winmm.lib
```

### 方法 B：MinGW

**前提**：安装了 MinGW（32位版本）

```cmd
build_mingw.bat
```

或手动：
```cmd
gcc -shared -o winmm.dll oni_winmm.c -lwinmm -luser32 -lkernel32 -O2 -m32
```

> ⚠ **必须是32位编译器**。游戏是32位PE，64位DLL无法注入。
> MSVC 用 "x86 Native Tools" 命令行，MinGW 用 `-m32` 参数。

## 安装到游戏

编译成功后会得到 `winmm.dll`，按以下步骤安装：

### 步骤 1：备份原始 winmm.dll

```
复制  C:\Windows\System32\winmm.dll
粘贴到  [游戏目录]\winmm_orig.dll
```

> 注意是 `System32`（不是 SysWOW64），因为32位程序会自动重定向到正确的目录。
> 实际上两个目录下的 winmm.dll 都可以用，DLL会自动查找。

### 步骤 2：放入劫持DLL

```
复制编译出的  winmm.dll
粘贴到  [游戏目录]\winmm.dll
```

### 最终游戏目录结构

```
[游戏目录]/
├── ONI.exe            ← 游戏主程序
├── ONI.INI            ← 游戏配置
├── oni.dat            ← 脚本数据
├── scene.dat          ← 场景数据
├── *.gr2              ← 图像资源
├── winmm.dll          ← 【你编译的dump DLL】
├── winmm_orig.dll     ← 【原始系统DLL备份】
└── _dump/             ← 【运行后自动生成】
    ├── dump_log.txt
    └── images/
        ├── 0001_sakura1.bmp
        ├── 0002_yn-11.bmp
        └── ...
```

### 步骤 3：运行游戏

正常双击 ONI.exe 启动游戏。游戏行为完全正常，但在后台会自动dump所有加载的图像。

### 步骤 4：查看结果

- **`_dump\images\`** — 所有dump出的BMP图像，文件名格式为 `序号_资源名.bmp`
- **`_dump\dump_log.txt`** — 详细日志，包含每张图的加载参数和尺寸

---

## 卸载

删除游戏目录下的 `winmm.dll`。

如果需要恢复原始状态，也可以把 `winmm_orig.dll` 改回 `winmm.dll`（或直接删掉，不影响游戏运行）。

---

## 原理说明

### 为什么用 winmm.dll 劫持？

Windows 加载 DLL 时优先搜索程序所在目录。ONI.exe 导入了 `WINMM.dll` 的3个函数（`mciSendCommandA`、`PlaySoundA`、`timeGetTime`）。我们在游戏目录放一个同名 DLL，Windows 就会加载我们的版本而非系统的。

我们的 DLL 做两件事：
1. **转发**：把3个winmm函数的调用转发给真正的系统 winmm.dll（改名为 `winmm_orig.dll`）
2. **Hook**：修改游戏图像加载函数 `FUN_0040C1E0` 的入口为 `JMP` 到我们的代码

### Hook 机制

```
游戏调用 FUN_0040C1E0("yn-11")
  │
  ├─ 被 JMP 劫持到 HookedImageLoad()
  │    │
  │    ├─ 记录日志
  │    ├─ 临时恢复原始字节
  │    ├─ 调用真正的 FUN_0040C1E0("yn-11")  ← 游戏正常加载图像
  │    ├─ 重新安装 JMP hook
  │    ├─ 从 0xBB7300 读取 BITMAPINFOHEADER（宽高、bpp）
  │    ├─ 从 0x429B50 读取解码后的像素数据
  │    └─ 保存为 BMP 文件
  │
  └─ 返回给游戏（游戏完全无感知）
```

## 故障排除

| 问题 | 解决 |
|------|------|
| 游戏启动时弹出"无法加载winmm.dll"错误 | 确认 `winmm_orig.dll` 存在于游戏目录 |
| 游戏正常运行但 `_dump` 目录为空 | 检查 `_dump\dump_log.txt` 看hook是否成功安装 |
| dump_log 显示 "Hook FAILED" | 可能有杀毒软件拦截了 VirtualProtect，添加游戏目录到白名单 |
| dump出的BMP图像是黑色/乱码 | 检查游戏EXE版本是否匹配（地址是硬编码的） |
| 编译报错"无法解析的外部符号" | 确保用的是32位编译器，且 link 了 winmm.lib |
