# 第一章　分析起点——字符串搜索与引擎识别

逆向工程的第一步是在陌生的可执行文件中寻找线索。游戏目录下有 `main.sct` 和 `scene.sct` 两个脚本文件，用十六进制编辑器打开可以看到文件头的魔数。

## 1.1 魔数识别

main.sct 的前 4 字节为 `4D 53 43 0A`，即 ASCII "MSC\n"。这是 MAI 引擎（也称 HS 引擎）的脚本封包格式签名。在 Ghidra 中搜索 "MSC"，立即命中文件加载函数。

## 1.2 文件加载入口

顺着 SCT 加载函数的调用链，定位到引擎初始化函数。该函数依次加载：

```
FUN_00421520 → 加载 main.sct（主脚本，230个场景）
FUN_00421520 → 加载 scene.sct（场景脚本，59个场景）
```

同时发现 AT 文件（选择支追踪数据）的加载路径，以及 VM 主循环 `FUN_00421710` 的入口。

## 1.3 关键发现

通过搜索 `##`（文本终止标记），确认游戏对话文本以 `##` (0x23 0x23) 结尾。这是 MAI 引擎的文本格式特征，与标准的 `\0` 终止不同。文本编码为 Shift-JIS (cp932)。

# 第二章　SCT 文件头与 Chunk 索引结构

## 2.1 文件头格式

SCT 文件以 32 字节的文件头开始：

| 偏移 | 大小 | 含义 | main.sct 实例 |
|---|---|---|---|
| 0x00 | 4B | 魔数 "MSC\n" | `4D 53 43 0A` |
| 0x04 | 4B | 文件总大小 | 0x130943 (1,247,555B) |
| 0x08 | 2B | 版本/标志 | 0x0001 |
| 0x0A | 2B | 全局 BC 段大小 | 0x1A47 (6,727B) |
| 0x0C | 4B | 未知字段 | 0x0CE00000 |
| 0x10 | 16B | 保留区 | 全零 |

文件头之后是 Chunk 区域，包含资源索引表。

## 2.2 Chunk 结构

每个 Chunk 以 8 字节头开始：

```
[type: uint32 LE] [count: uint32 LE]
```

后跟 `count` 个固定大小的记录。Chunk 按类型顺序排列：

| Type | 含义 | 记录大小 | main.sct 数量 |
|---|---|---|---|
| 0 | 声音资源 | 37B | 194 |
| 1 | 图形资源 | 48B | 517 |
| 3 | 场景表 | 35B | 230 |

Type 3 Chunk 之后紧跟**全局 BC 段**（大小由 header +0x0A 指定），再之后是**场景数据区**——230 个场景的二进制数据依次排列，总计占据文件的绝大部分空间。

## 2.3 全局 BC 段

全局 BC 段是一块共享数据区，存储跨场景的全局变量和常量。VM 指令通过高位标志（`val | 0x80000000`）区分全局引用和场景内引用：

```c
// FUN_004221a0 地址解析
if (val & 0x80000000)
    return global_bc_base + (val & 0x7FFFFFFF);  // 全局引用
else
    return scene_base + val;                       // 场景内引用
```

# 第三章　Type3 场景记录——隐藏的 f1/f2 字段

## 3.1 记录格式

每条 Type3 记录 35 字节，描述一个场景在文件中的位置和属性：

| 偏移 | 大小 | 字段 | 说明 |
|---|---|---|---|
| 0x00 | 17B | name | 场景名，ASCII，\0 填充 |
| 0x11 | 2B | u16 | 用途不明 |
| 0x13 | 4B | offset | 场景数据在文件中的绝对偏移 |
| 0x17 | 4B | size | 场景数据大小 |
| 0x1B | 4B | f1 | **状态变量区偏移** |
| 0x1F | 4B | f2 | **状态变量区大小** |

前四个字段的含义一目了然。但 f1 和 f2 的含义在初始分析中完全不明——直到变长注入测试时才揭示其重要性。

## 3.2 f1/f2 的引擎用途

通过 Ghidra 逆向 `FUN_00422230`（约 16,000 字符的大型函数），找到三处对 f1 的关键引用：

```c
// 读取：从场景数据的 f1 偏移处复制 f2 字节
puVar15 = (scene_data + f1);
for (i = f2 >> 2; i != 0; i--) { *dest++ = *puVar15++; }

// 写入：存档恢复时写回状态
pbVar3 = (scene_data + f1);
for (i = size >> 2; i != 0; i--) { *pbVar3++ = *src++; }
```

**f1 是场景内"状态变量区"的起始偏移，f2 是该区域的大小**。状态变量区位于指针表中（详见第五章），存储选择结果、游戏进度等运行时状态。引擎在存档/读档时通过 f1/f2 定位和操作这些变量。

## 3.3 f1 的验证

对全部 230 个场景验证：**f1 + f2 始终小于 first_text_pos**（第一个文本的偏移）。这意味着状态变量区完全位于文本区之前，不受文本替换影响——这是后续章节等长方案可行的关键前提。

# 第四章　场景内部的三段式数据布局

## 4.1 场景 Blob 总览

每个场景 blob 的内部结构可以分为三个主要区域：

```
┌──────────────────────────────────────┐
│ +0x00  [1B]  场景类型标记 (0x40)     │
│ +0x01  [4B]  bc_off (bytecode偏移)   │ ← 头部
├──────────────────────────────────────┤
│ +0x05  指针表 + 控制数据区           │
│         ↓ u32 文本指针               │
│         ↓ u32 状态变量 (0,1,8,FF)    │
│         ↓ u32 零填充                 │ ← 指针表区
├──────────────────────────────────────┤
│ +first_text  文本+字符串链           │
│         ↓ ##终止的对话文本            │
│         ↓ 选择支文本 (非##)           │
│         ↓ 语音标签 (_vMizu等)         │
│         ↓ 场景名引用 (_efBlack等)     │ ← 字符串链区
├──────────────────────────────────────┤
│ +bc_off  Bytecode 区                │
│         ↓ VM 指令流                  │ ← 代码区
└──────────────────────────────────────┘
```

## 4.2 以 Scene 47 (c_asa18) 为例

这是包含角色选择（牧瀬みずほ/桐沢天音/中久保ひなの）的场景：

```
场景大小: 0x16B0 (5808B)
bc_off:   0x08B3

指针表区: +0x05 ~ +0xDC   (54个u32: 42指针+1控制+11零)
字符串链: +0xDD ~ +0x8B2  (42个字符串: 20个##文本 + 22个选择支/标签)
代码区:   +0x8B3 ~ +0x16AF

f1=0xAD  f2=0x30 → 状态变量区 [+0xAD, +0xDD) 共48字节
```

## 4.3 以 Scene 25 (_zSelect) 为例

这是集中管理所有选择支的特殊场景，结构最复杂：

```
场景大小: 0x1B2B (6955B)
bc_off:   0x1687

指针表区: +0x05 ~ +0x604  (384个u32: 135指针+115控制+111零+23其他)
字符串链: +0x605 ~ +0x1686 (135个字符串: 42个##文本 + 93个选择支/标签)
代码区:   +0x1687 ~ +0x1B2A

f1=0x5C9  f2=0x3C → 状态变量区 [+0x5C9, +0x605) 共60字节
```

其中 93 个非##字符串包括所有路线的选择支文本（如"面白いです""納得できない"等）、语音标签（_vMizu、_vAman 等）和场景名引用。

# 第五章　指针表的秘密——指针、控制数据与字符串链

这是整个逆向分析中最关键的发现。

## 5.1 指针表不是纯指针数组

初始假设：指针表区域（+0x05 到 first_text）全部是 u32 文本指针。但实际分析发现，指针表包含三种不同类型的数据：

| 类型 | 值范围 | 含义 |
|---|---|---|
| 文本指针 | `[first_text, bc_off)` | 指向字符串链中某个字符串的起始 |
| 状态变量 | 0, 1, 8, 0xFF 等小值 | 运行时可修改的游戏状态标志 |
| 零填充 | 0x00000000 | 未使用的槽位 |

这三种数据混排在同一个 u32 数组中。引擎通过 f1/f2 字段定位状态变量区的范围。

## 5.2 字符串链的连续性

**关键发现：指针表中的文本指针值严格递增，且指向的字符串紧密排列、零间隙。**

对所有 230 个场景验证：

- 每个指针值恰好对应一个 `\0` 终止字符串的起始位置 ✓
- 没有任何字符串起始位置不被指针引用 ✓
- 相邻字符串之间仅隔一个 `\0` 字节，无额外间隙 ✓
- 整条链从 first_text 一直延续到 bc_off ✓

这意味着字符串链 = `first_text` 到 `bc_off` 之间的连续空间，由指针表中的指针值索引。

## 5.3 ##文本与非##字符串的交替排列

一个重要的教训：**##文本和非##字符串不是分开存放的，而是交替排列的。**

以 Scene 47 的字符串链为例：

```
+0x00DD [##] 「しかし……みずほさんの面影が脳裏に……」##
+0x011E [##] 「ほかのふたりのことも気になるけど……」##
  ...
+0x05A4 [##] 「ほかでもない、彼女の面影が……。」##
+0x05C7 [选] ふたりに集中しよう           ← 选择支文本（非##）
+0x05DA [选] ひなのさんも気になる
  ...
+0x063D [选] 牧瀬みずほ                   ← 角色名选项
+0x0648 [选] みずほさんに集中しよう
  ...
+0x06A1 [##] 「おっと、待てよ？……」##     ← 又是##文本
+0x0787 [选] みずほさんだけでいい          ← 又是选择支
  ...
```

这种交替排列意味着：**不能简单地"把所有##文本紧密排在前面"——这会覆盖中间的选择支文本。**

## 5.4 字符串链是"章节"的核心

从汉化的角度看，字符串链就是一个"章节"：

- 总大小固定（= bc_off - first_text）
- 内部元素可以重排（只要更新指针值）
- ##文本可以变长/变短，只要总量不超过链的容量
- 非##字符串的内容必须保持不变

这就是章节等长注入方案的理论基础。

# 第六章　VM 指令集逆向

## 6.1 VM 架构

MAI 引擎的 VM 是一个寄存器机，核心状态存储在 VM 对象中：

| 偏移 | 字段 | 含义 |
|---|---|---|
| +0x11C | PC | 程序计数器（场景内偏移） |
| +0x120 | scene_size | 当前场景数据大小 |
| +0x124 | scene_base | 当前场景数据缓冲区指针 |
| +0xD4 | global_base | 全局 BC 段基址 |
| +0xDC~+0x118 | regs[16] | 16 个通用寄存器 |

## 6.2 指令格式

VM 指令有五种格式，通过 opcode 字节的位模式区分：

```
跳转指令 (opcode & 0xFE ∈ {0x40, 0x42}):
  [op:1B] [target:int32]                    = 5B

特殊指令 (opcode & 0xFE == 0xA0):
  [op:1B] [sub:1B] [val:u32]               = 6B

算术A (opcode & 1 == 1):
  [op:1B] [reg1:1B] [reg2:1B]              = 3B

算术B (opcode & 1 == 0, flags & 0x80 == 0):
  [op:1B] [flags:1B] [val1:u32] [val2:u32] = 10B

算术C (opcode & 1 == 0, flags & 0x80 != 0):
  [op:1B] [flags:1B] [val:u32]             = 6B
```

## 6.3 跳转与场景切换

**跳转指令**（0x40 条件 / 0x42 无条件）的 target 是场景内绝对偏移，始终指向 bytecode 区（≥ bc_off）。所有跳转均为前向跳转。

**场景切换**通过 0xA0 特殊指令触发。VM 从指针表读取目标场景名，调用 `FUN_00431af0` 加载新场景。场景加载过程：

1. 在 Type3 表中按名称匹配目标场景
2. 如果场景数据未加载，分配缓冲区并从文件读取
3. 设置 scene_base、scene_size、PC=0

## 6.4 地址解析

算术B/C 格式中的 u32 操作数通过 `FUN_004221a0` 解析为绝对地址：

- 高位为 0：`scene_base + val` → 引用当前场景内数据
- 高位为 1：`global_base + (val & 0x7FFFFFFF)` → 引用全局 BC 段

场景内引用的值可能指向指针表区（控制数据操作）或字符串链区（文本读取）。

# 第七章　AT 选择支追踪系统

## 7.1 AT 文件格式

AT 文件记录玩家的选择历史，用于 CG 解锁、结局判定等。格式如下：

```
[0x00] uint32  场景数量 (= 230)
[0x04] uint32  文件大小
[0x08] 场景条目 × 230 (每条 25 字节)
[...]  参数数据区 (每组 5 字节)
```

## 7.2 条目与参数

每条场景条目：

| 偏移 | 大小 | 含义 |
|---|---|---|
| 0x00 | 17B | 场景名 |
| 0x11 | 4B | 参数数量 n_params |
| 0x15 | 4B | 参数数据偏移 data_off |

每个参数 5 字节：`[value:u32] [flag:u8]`

- value：场景内 bytecode 区的偏移（AT 触发点位置）
- flag：0=未访问，1=已访问

## 7.3 引擎行为

`FUN_00423330`（AT 搜索）：每次 VM 执行到某个 PC 位置时，检查是否匹配某个 AT 参数。如果匹配，标记该参数为"已访问"。

`FUN_00423370`（AT 查询）：检查某个 AT 参数的访问状态，返回给调用者。

**重要：AT 系统不控制跳转逻辑——它只做记录和查询。** 选择分支的实际控制由 bytecode 中的条件跳转完成。

# 第八章　文本注入方案演进——从等长到章节等长

## 8.1 等长模式（初始方案）

最简单的注入方式：每条翻译文本填充到原始文本的精确大小。短则补空格 (0x20)，长则截断。

**优点：** 完全不改任何偏移，绝对安全
**缺点：** 163 条文本被截断，信息损失

## 8.2 变长模式（完整偏移修正）

对场景内所有引用进行六层修正（详见第九章）。理论上完美，所有静态验证通过。

**结果：** Scene 25 (_zSelect) 在选择支跳转时崩溃。等长模式正常，变长模式崩溃——即使只多 1 个字节。问题锁定在引擎内部对 bc_off 或 scene_size 的某种隐藏依赖，需要动态调试才能定位。

## 8.3 章节等长模式（最终方案）

核心思想：**利用字符串链的紧密排列特性，在固定大小的链内重新分配空间。**

原理：

1. 每个场景的字符串链（first_text → bc_off）大小固定
2. 翻译后的中文普遍比日文短，链内总量有富余
3. 把所有元素（##文本 + 非##字符串）按顺序紧密重排
4. ##文本用翻译替换，非##字符串保持原内容
5. 短文本省下的空间自动匀给长文本
6. 只更新指针表中的指针值

**不需要修改的东西：** bc_off、场景大小、AT 参数、bytecode 跳转目标、文件大小、f1/f2 字段——全部保持原值。

**验证结果：**

```
main.sct:  7671 条替换，0 截断，大小不变
scene.sct: 4863 条替换，0 截断，大小不变
总计: 12534 条，100% 完整保留，零信息损失
```

## 8.4 为什么章节等长能做到零截断？

中文翻译的 cp932 编码通常比日文原文短：

- 日文平假名/片假名：每个字符 2 字节
- 日文汉字：每个字符 2 字节
- 中文翻译使用 cp932 中的汉字：每个字符 2 字节
- 但中文表达通常更简洁，字符数更少

统计显示，每个场景内翻译后的总字节数均小于原文总字节数。字符串链的容量（= bc_off - first_text）足以容纳所有翻译。

# 第九章　变长注入的六层修正与未解之谜

虽然变长模式最终未能完美工作，但其分析过程揭示了引擎的深层结构。

## 9.1 六层修正体系

```
① 场景表 (type3 record offset/size)
② AT 参数 (value += cum_delta)
③ 指针表 (u32 文本指针，逐段 delta)
④ bc_off (场景内 bytecode 偏移)
⑤ Bytecode 跳转目标 (target += cum_delta)
⑥ Bytecode 文本区操作数 (逐段 delta)
```

每一层都经过了独立验证。13722 个修正后的指针全部通过内容一致性检查。

## 9.2 指针表扫描边界 Bug（已修复）

初始实现中，指针表扫描范围为 `j < new_bc_off`，会扫描到文本区内部。SJIS 文本字节被读作 u32 值，部分恰好落在修正范围内，导致**文本内容被破坏**。

修复：限制扫描范围为 `j < first_text_pos`——只扫描真正的指针表区域。

## 9.3 未解之谜

所有静态验证通过，但 Scene 25 在变长模式下仍然崩溃。崩溃地址 0x421b06，scene_base = NULL，异常码 0xC0000005。

已排除的原因：

- 指针修正错误（13722 个全部正确）
- 文本内容损坏（逐字节验证）
- AT 参数错误（7808 个正确）
- 跳转目标错误（224 个正确）
- f1/f2 需要更新（不需要，f1+f2 < first_text）
- 非##字符串损坏（6273 个全部正确）

疑似原因：引擎在 `FUN_00422230` 中进行跨场景数据传递时，可能计算了依赖于 bc_off 或 scene_size 的某个中间值，文本变长后该值与预期不符，导致后续场景加载失败。需要 x64dbg 动态调试来定位。

# 第十章　验证方法论与踩坑记录

## 10.1 Identity Round-Trip

将原始文本作为"翻译"输入，验证输出与原始文件 MD5 完全一致。这是最基本的工具正确性检查——如果"翻译"没有改变任何文本，输出就不应该有任何字节变化。

## 10.2 最小变更隔离

当问题无法定位时，逐步缩小变更范围：

| 测试 | 结果 | 结论 |
|---|---|---|
| 只改 1 个字节 | 崩溃 | 不是累积效应 |
| 只改最后一个场景 | 正常 | 问题在特定场景 |
| 等长替换同一位置 | 正常 | 问题在偏移变化 |
| 手动二进制修改 | 正常 | 排除工具 bug |

## 10.3 控制流反汇编

VM 指令长度不固定（3/5/6/10 字节），线性扫描会错位。通过控制流追踪——从入口点开始，沿跳转目标和顺序执行路径遍历——可以正确识别所有可达指令。

## 10.4 踩坑记录

**坑 1：指针表扫描进文本区**

`while j < new_bc_off` 扫到了文本区，把 SJIS 字节当 u32 指针修正。修复：`while j < first_text_pos`。

**坑 2：find() 验证误报**

用 `find()` 查找字符串位置来验证指针时，同名字符串出现多次导致误判。例如 `_grph` 出现两次，`find()` 返回第一个而指针指向第二个。应直接比较 `scene[ptr_value:]` 处的内容。

**坑 3：字符串链紧密重排覆盖非##文本**

初版章节等长代码将所有##文本紧密排列，覆盖了中间夹杂的选择支文本。修复：把##和非##字符串作为统一的链元素处理，按指针顺序重排。

**坑 4：bytecode 操作数也可能引用文本区**

最初只修正了 bytecode 中的跳转目标（层⑤），遗漏了算术指令中引用文本区的 u32 操作数。全文件共 128 个操作数需要修正（层⑥）。

```bash
# 章节等长模式（推荐：0截断，100%安全）
python sct_inject.py --main main.sct main.json \

    --at at --outdir output --chaplen

# 逐条等长模式（安全但有截断）
python sct_inject.py --main main.sct main.json \
    --scene scene.sct scene.json \
    --at at --outdir output

# 变长模式（Scene 25 有已知问题）
python sct_inject.py --main main.sct main.json \
    --scene scene.sct scene.json \
```
